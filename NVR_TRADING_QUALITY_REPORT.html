What's Excellent

AI Decision Prompt — Physics-Based, Not Sentiment-Based

The system prompt follows a clear hierarchy: Capital Flow → Price Action → On-Chain Flow → Technical Indicators. Fear & Greed index was intentionally demoted to info-only (v20.8), showing active refinement from real trading experience. The prompt is ~450 tokens (core) + ~800 tokens (strategy) — not bloated. Output is strict JSON, no prose parsing needed.

Swarm Intelligence — Fast, Math-Only Consensus

Five micro-agents (momentum, trend, flow, risk, sentiment) evaluate tokens using pure math — no Claude API calls. Weighted vote aggregation produces consensus + confidence in <100ms for 20+ tokens. Optional activation means zero cost when markets are calm.

Data Pipeline — 8+ Sources, Parallel, Fault-Tolerant

Market data from CoinGecko, DeFiLlama, FRED, Binance, GeckoTerminal, DexScreener, CryptoPanic, and on-chain RPC — all fetched in parallel with fallback chains. Phantom spike detection catches bad price data. On-chain P&L uses Blockscout as source of truth.

Three Priority Improvements

These are the highest-impact quality improvements that fit cleanly into the new architecture. Each is a targeted change, not a new feature. Together they protect against the worst-case scenarios when scaling to hundreds of users.

1. Trailing Stop Hold Lock — The Meme Coin Bleeder HIGH PRIORITY

The Problem: MIN_HOLD_CYCLES = 4 locks every position for 1 hour before any stop-loss can fire. This prevents buy-then-sell churn (good), but on meme/AI tokens, a flash crash can eat 15-20% before the stop is even allowed to trigger.

Real Scenario: Buy $100 of a meme token. It crashes 15% in 2 cycles (30 min). Stop-loss is calculated at -8%, but the hold lock prevents it from firing for 2 more cycles. By then, position is -20%. The stop finally triggers, but the damage is done.

Evidence: Sector stop-loss overrides exist for memes (-4% max, -3% trailing), but they only tighten the calculation, not the hold-time lock.

The Fix (3 lines in config/constants.ts):

// Reduce hold lock for high-risk sectors
SECTOR_MIN_HOLD_CYCLES: { 'Meme': 2, 'AI': 2, default: 4 }

// Emergency override: bypass hold lock if loss exceeds threshold
EMERGENCY_STOP_LOSS_OVERRIDE_PCT: 5  // If down >5%, stop fires immediately

Impact: Prevents the single biggest loss scenario for new users. Meme positions can still be held for conviction, but won't silently bleed to -20%.

Effort: LOW — Config change + 5-line check in stop-loss evaluation

2. Self-Improvement Overfitting Guard — The Bot That Broke Itself HIGH PRIORITY

The Problem: The self-improvement engine adapts confluence thresholds based on recent trade performance. If all recent trades happen in one market regime (e.g., trending up), it over-optimizes for that regime and gets crushed when markets flip.

Evidence (from the codebase):

// v21.2: capped at 28 (was 30) — bot tuned itself to 30
// and stopped trading entirely

This comment proves the bot already broke itself once. The cap at 28 is a band-aid. The lower bound of 5 means the bot could also make entries absurdly easy in a winning streak, then hemorrhage when regime changes.

Current Guards (insufficient):

  • MIN_REGIME_DIVERSITY = 2 — exists but only checked at promotion, not during confirmation queue
  • MAX_CONTRADICTION_RATIO = 0.3 — needs 30% failures to reject, which takes 10+ trades
  • Bounds [5, 28] prevent extremes but allow wide oscillation within bounds

The Fix (in src/self-improvement/engine.ts):

// 1. Lock adaptation velocity
MAX_ADAPTATIONS_PER_24H = 2  // Prevent spiral (was unlimited)

// 2. Require regime diversity in confirmation queue
// Shadow proposals must see 2+ distinct regimes BEFORE promotion
// (Currently only checked at final promotion step)

// 3. Tighter bounds per regime awareness
// confluenceBuy range: [8, 20] (was [5, 28])

Impact: Prevents the bot from paralyzing itself or becoming recklessly aggressive. Critical for user trust.

Effort: MEDIUM — 15-20 lines in self-improvement engine + config constants

3. Kelly Fallback Ramp — Protect New Users MEDIUM PRIORITY

The Problem: Before 20 trades, Kelly criterion has no data and falls back to 8% of portfolio per trade. On a $1,000 account, that's $80 per trade. If the first 5 trades all lose, that's $400 gone (40% of capital) before Kelly even has enough data to calculate properly.

Why This Matters for Scaling: New users deploying their first bot will experience 5-10 trades with zero historical performance data. These early trades determine whether they trust the system or abandon it. Aggressive sizing in this window is the #1 onboarding risk.

The Fix (1 line in src/algorithm/position-sizing.ts):

// Ramp up from conservative to full size as history builds
const successfulTrades = state.tradeHistory.filter(t => t.success).length;
const ramp = Math.min(1.0, successfulTrades / 20);
fallbackSize = fallbackSize * Math.max(0.15, ramp);
// Trade 1: ~$12 (15% of $80)
// Trade 5: ~$20 (25% of $80)
// Trade 10: ~$40 (50% of $80)
// Trade 20: $80 (full size)

Impact: A new $1,000 bot that loses its first 5 trades loses ~$75 instead of ~$400. User retains capital and confidence while the bot calibrates.

Effort: LOW — 3 lines in position sizing module

Secondary Findings

These are real issues but lower priority than the three above. Documented for future sessions.

Cost Basis Sanity Reset

When cost basis exceeds 20x current price, it resets to market price. This can orphan realized P&L history. Fix: preserve realizedPnL separately before any reset, require 2 consecutive cycles of bad data before resetting.

EFFORT: LOW

Kelly Undersample in Trends

If last 50 trades are all BUYs (trending market, no exits yet), Kelly sees 0 sells in its rolling window and falls back to aggressive default. Fix: extend sell lookup to include sells related to buys within the window.

EFFORT: MEDIUM

TWAP Slippage Stacking

Orders >$100 split into 5 TWAP slices with 12s delays. In high volatility, later slices can hit dramatically different prices. Fix: adjust slice count dynamically based on current ATR (fewer slices in calm, more in volatile).

EFFORT: MEDIUM

Risk Reviewer Misses Correlation

Checks single-position limits and sector balance, but doesn't detect when 5 positions are all "AI tokens" (high correlation). Fix: add deterministic sector-overlap detection that warns when 3+ positions share a sector.

EFFORT: LOW
